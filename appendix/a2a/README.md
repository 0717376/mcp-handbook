# Appendix — A2A case study

Сравнение MCP с **A2A (Agent-to-Agent Protocol)** — протоколом от Google для общения между агентами. Смысл раздела: A2A и MCP решают **разные задачи**, но частично пересекаются, и на этом пересечении видно, где MCP концептуально слаб и куда он движется.

Темы:

- **Модель взаимодействия.** MCP — sync RPC с notifications (запрос → ответ, опциональные прогресс-уведомления по ходу). A2A — task-oriented с durable task handles: «запусти задачу, вот тебе id, приходи потом за результатом».
- **Discovery.** В MCP клиент знает адрес сервера и capabilities узнаёт в `initialize`. В A2A есть Agent Card — стандартный формат метаданных агента в `.well-known/`, позволяющий другим агентам находить и понимать друг друга. MCP движется в ту же сторону через **Server Cards SEP**.
- **Long-running operations.** В MCP нет нативного понятия «задача, которая живёт дольше соединения». Progress notifications — это опциональная приправа к синхронному вызову. В A2A задача — это first-class сущность с жизненным циклом (submitted / working / input-required / completed / failed / canceled). Именно это **SEP-1686 Tasks** пытается принести в MCP.
- **Cancellation и resume.** В MCP cancellation — notification, resume между соединениями не определён. В A2A это часть протокола by design.
- **Auth.** Оба используют OAuth 2.1. A2A сразу проектировался multi-agent, MCP добавил OAuth позже, в 2025-06-18.
- **Где граница.** MCP — это «агент/host ↔ инструменты/данные». A2A — это «агент ↔ агент». В сложной системе, вероятно, увидишь оба: внутри агента — MCP к внешнему миру, между агентами — A2A (или MCP с Tasks primitive, когда он стабилизируется).

Опирается на §3 (lifecycle, примитивы), `examples/06-notifications/` и `examples/07-cancellation/` (чтобы было с чем сравнивать). Читается после основной прогрессии.
